Every upside & every downside

1
Metadata / field filter
date · entity · folder · tags · status
the secret weapon for "yesterday"

Upsides

  • Sub-millisecond — indexed field lookup (B-tree/hash), not a scan.
  • Scales to billions — this is exactly what databases do.
  • Exact & deterministic — "yesterday", date ranges, a specific client, a folder.
  • Collapses the corpus first — narrows 1M notes to ~3 before any heavy search runs.
  • No model, no network, fully offline, trivial compute.
  • Composable — AND/OR across fields (date AND folder AND entity).
  • Always-current if fields are indexed on write.
  • Fully explainable — you know exactly why a note matched.

Problems

  • Only as good as your metadata — needs date/entity/tag fields to exist and be accurate; missing/garbage fields = useless.
  • It's a filter, not a search — can't find anything by free-text content, and doesn't rank by relevance on its own.
  • Needs an extraction/tagging pipeline (or entity detection) to populate fields reliably.
  • "Yesterday" ambiguity — created vs modified vs event date; needs a clear timestamp convention.
  • Schema rigidity — a new filter dimension means a new indexed field + reindex.
  • Can't interpret nuance ("the difficult client") unless it's literally a field.
  • Index maintenance on every write/update/delete.
Best for: time/entity/category queries — the front filter of the pipeline.NV today: folder filter exists; date/entity fields not yet indexed.
2
Lexical / BM25 (TF-IDF)
tokenized inverted index
NV's current default + fallback

Upsides

  • Fastest ranked content search — µs–ms hash lookup, no embed step.
  • Relevance-ranked (TF-IDF/BM25) — best-first, with snippets.
  • Scales to billions — Lucene/Elasticsearch territory.
  • No model/network — works offline; the reliable cold-start fallback.
  • Tiny memory vs vectors; cheap, fast index build.
  • Strong exact term/name/ID match (client names, codes).
  • Deterministic & explainable (you see the matched terms).
  • Multi-word scoring; phrase/proximity possible with a positional index.

Problems

  • Whole-token only — misses typos & partials (auth ≠ authentication) unless stemmed.
  • No semantics — misses synonyms/paraphrase (car ≠ automobile); the core weakness for conversational questions.
  • Vocabulary mismatch — query words must overlap the note's words.
  • Stopword/short-token dropping can lose meaning (NV drops <2-char + stopwords).
  • Bag-of-words — ignores word order/meaning unless positional.
  • Index can go stale — needs rebuild/sync (NV rebuilds at launch).
  • NV-specific: current impl allocates an O(N) score array per query — fine now, must become postings-only for millions.
  • Stemming/lemmatization is language-specific and adds complexity.
Best for: exact keywords/names, ranked — the everyday workhorse.NV today: implemented (TF-IDF). qmd uses BM25.
3
Trigram / n-gram
character 3-grams · fuzzy substring
pg_trgm / ripgrep / Zoekt

Upsides

  • Typo-tolerant & partial — Jonh→John, auth→authentication, half-remembered names.
  • Substring & mid-token matches anywhere in the text.
  • Language/script-agnostic — great for code, IDs, non-English, autocomplete.
  • Fast — in-memory set intersection of trigram lists; no model/network, offline.
  • Fills the exact gap grep (no typos) and BM25 (whole-token) both miss.

Problems

  • Big index, high memory — every 3-gram is a key (~3–5× the inverted index); needs disk-backing past ~100k.
  • False-positive candidates — needs a verify/rank pass; weak native relevance ranking.
  • Not semantic — no synonyms/meaning.
  • Queries <3 chars are awkward/ineffective.
  • Higher build & update cost than a plain inverted index.
  • Common trigrams (the/ing) get noisy — needs tuning/weighting.
  • Lower precision than exact/BM25 on already-clean queries.
Best for: misspelled / partial names, code & identifier search.NV today: not implemented — the recommended add for fuzzy name matching.
4
Vector / Semantic (ANN)
embeddings + HNSW / LanceDB
the only "understands meaning" tier

Upsides

  • Semantic — matches meaning, synonyms, paraphrase, cross-lingual; answers conversational queries with no shared keywords.
  • ANN retrieval ≈ O(log N) — sub-ms even at millions/billions (HNSW/FAISS/LanceDB).
  • Robust to vocabulary mismatch — no exact words needed.
  • Scales with disk-backed stores (LanceDB) + quantization.
  • High recall for vague/underspecified questions.

Problems

  • Query-embed latency ~1.2s (measured) — the dominant per-query cost; every query pays it unless cached.
  • Brute force is O(N·768) — breaks past ~tens of thousands (NV's current impl); must move to HNSW/LanceDB.
  • Memory/disk heavy — 768-dim f32 ≈ 300MB@100k, 3GB@1M → needs int8 quantization or disk-backing.
  • ANN is approximate — recall <100%; can miss the true nearest (tunable ef/M, speed↔recall tradeoff).
  • Weak on exact tokens — IDs, codes, names; can sound right but be wrong (inverse of lexical).
  • Embeddings go stale — must re-embed on edit; new notes invisible until embedded (NV embeds at launch only).
  • Expensive index build (embed every note); a model change = full reindex.
  • Hard to explain a ranking; chunking long notes affects quality; embedder quality caps the ceiling.
  • Needs a model present (offline only if the embedder is local).
Best for: natural-language / conceptual questions (Cortex's core need).NV today: brute-force cosine, in-memory vectors.json; LanceDB is the planned M2 upgrade.
5
Hybrid + reranker
RRF (lex+vec) → cross-encoder
best accuracy, highest cost

Upsides

  • Best accuracy — exact (lexical) + semantic (vector) cover each other's blind spots.
  • RRF is normalization-free, simple, robust, scalable (k=60 default — NV uses this).
  • Reranker (cross-encoder on top ~50) gives top-tier final precision.
  • Scales — every component scales; rerank is constant cost on top-k only.
  • Graceful degradation — fall back to lexical if the embed times out.

Problems

  • Slowest — pays embed + both searches + fusion + a second (reranker) model.
  • Most complex to build, tune, and maintain (two indexes + fusion + reranker).
  • Reranker = another model dependency + latency (tens–hundreds of ms on top-k).
  • More moving parts → more failure modes.
  • Overkill for simple exact lookups.
  • Tuning burden — RRF k/weights, rerank cutoff, candidate pool sizes.
  • Highest overall resource cost (CPU/RAM/disk).
Best for: when the answer must be exactly right and the query is conversational.NV today: hybrid (RRF) implemented; reranker not yet. Embed 600ms budget < 1.2s real → often degrades to lexical.

★ Recommended pipeline for "which client did I talk to yesterday?"

1
Metadata filter — date = yesterday (+ folder/entity). Sub-ms, scales to millions → collapses the vault to ~3 notes.
2
Hybrid (lexical + vector-ANN, RRF) — run only on what's left (basically free). Catches exact names AND conversational phrasing.
3
Reranker — order the top few; hand top ~5 to Cortex.
4
Stream a fast LLM — the answer-writing is the only real wait; streaming + a fast model makes it feel instant.
Because step 1 narrows the corpus before any embedding or LLM work, vault size (10k → 1M) barely changes the answer time. Make NV scale by: raise the 600ms embed budget → ~2s · faster/quantized query embedder · HNSW/LanceDB instead of brute-force cosine · int8 vectors · add date/entity metadata fields + a reranker.
